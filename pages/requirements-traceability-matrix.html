Requirements traceability matrix

Document metadata

  • Document type: Project execution page
  • Status: Working baseline for delivery control
  • Scope: Methodological framework plus proof-of-concept demonstrator for AI-enabled SMCE support in JRC Unit S.3
  • Primary sources:

    • - AI-enabled SMCE technical proposal - Next steps after contract reread

  • Contract completion date referenced in source material: 22 May 2026

What this page is for

  • Turns the contract and proposal commitments into a delivery-control matrix.
  • Maps each requirement to an operational interpretation, a proposed project response, the evidence/artifact that should prove compliance, and any unresolved issue.
  • Acts as the spine for review meetings, internal QA, and final package assembly.

How to use this matrix

  • Treat each row as a control item that must be evidenced before final submission.
  • Update the Open issue / decision needed column whenever JRC guidance, scope, or technical choices change.
  • Link completed rows to the actual artifact versions used in the demonstrator package.
  • Use this page together with AI intervention-point register and Human oversight and approval gates.

Project-level control notes

  • The assignment is methodology-first, not production-system-first.
  • The proof of concept is illustrative and synthetic, not a live EC operational deployment.
  • Final analytical judgement, scoring, trade-offs, and policy decisions remain human responsibilities.
  • Every important AI-assisted output should remain identifiable as draft until approved.
