Next steps after contract reread

Why these are the next steps

The contract/proposal is clear that the assignment is not to build a production AI system first. It is to deliver one integrated package containing: 1. a documented methodological framework, and 2. a proof-of-concept demonstrator.

The contract logic is governance-first, methodology-first, and review-first. The next work should therefore reduce ambiguity in intervention points, oversight, evidence handling, logging, review roles, and demonstrator evaluation.

What the reread confirms

Key contractual/requirements signals: - Deliver one integrated deliverable in English and electronic form. - Complete by 22 May 2026. - Work in close coordination with JRC Unit S.3. - Build explicit AI intervention points across the workflow. - For each intervention point, specify: - what AI may do, - what AI may not do, - who reviews, - what record is kept. - Keep uncertainty, disagreement, and missing evidence visible. - Keep prompts/actions, revisions, and approvals logged. - Use a synthetic use case, not a live EC file. - Keep final judgement, scoring, trade-offs, and policy decisions with humans. - Align with Better Regulation, traceability, transparency, oversight, and auditability.

Proposed next work packages

1. Lock the scope in a requirements traceability matrix

Create a matrix from contract requirement -> design response -> evidence of compliance.

Why first: - prevents scope drift - makes review easier - gives JRC a clear compliance map

Minimum columns: - requirement / clause - interpretation - proposed response - artifact that satisfies it - open issue / decision needed

2. Define the AI intervention-point map

Create the operational backbone of the method.

For each intervention point, define: - stage of workflow - input - allowed AI task - prohibited AI task - human reviewer role - output artifact - logging requirement - quality check - escalation rule if evidence is weak or contested

Likely intervention points: - source ingestion / evidence-sheet drafting - criteria-structure drafting - rationale-note drafting - gap/inconsistency flagging - review-summary drafting

3. Define human roles and approval gates

The proposal repeatedly says “human reviewer,” but this still needs operational specification.

Define at least: - analyst / preparer - methodological reviewer - policy owner / case lead - validation / sign-off role

For each gate, specify: - what is being approved - approval criteria - what happens if rejected - what gets logged

4. Design the core artifact set

The methodology should probably be built around a small controlled set of standard artifacts.

Recommended artifacts: - requirement traceability matrix - evidence sheet template - criteria-structure template - rationale-note template - uncertainty register - intervention-point log - review-and-approval log - demonstrator test script - final internal summary template

5. Define the synthetic use case now

Do not leave the demonstrator example vague.

Specify: - policy domain - option set - affected groups - sample evidence corpus - intended criteria structure - uncertainty features to test - review checkpoints - success conditions

The synthetic case should be rich enough to test: - incomplete evidence - contradictory evidence - distributional effects - stakeholder disagreement - need for human override

6. Set demonstrator evaluation criteria

The proposal promises a demonstrator, but evaluators will want to know how it is judged.

Recommended evaluation dimensions: - traceability - usability for officials - completeness of logs - clarity of human oversight - ability to preserve uncertainty/disagreement - quality of structured outputs - time saved vs manual process - failure modes and safe handling

7. Write the governance rules as an auditable protocol

Turn the governance claims into short testable rules.

Example rule structure: - rule - rationale - applies to which intervention points - mandatory evidence/log - reviewer responsibility - non-compliance response

8. Decide the minimum technical architecture for the proof of concept

Because the contract is not asking for full production integration, pick the lightest architecture that still proves the method.

Likely correct direction: - local or controlled environment - file-based artifacts - transparent logs - no unnecessary systems integration - easy replay of prompts/actions

9. Prepare the final integrated deliverable structure early

Since the contract requires one integrated deliverable, define the package now.

Suggested final structure: - Part A: methodological framework - Part B: governance and oversight protocol - Part C: artifact templates - Part D: synthetic use case - Part E: demonstrator walkthrough - Part F: evaluation results and recommendations - Annexes: logs, sample artifacts, traceability matrix

Immediate recommendation

The single best next step is:

Create a requirements traceability matrix plus intervention-point register.

That gives the project its control spine and lets every later piece map back to contract requirements.

Suggested execution order

  1. Requirements traceability matrix
  2. Intervention-point register
  3. Human roles and approval gates
  4. Artifact templates
  5. Synthetic use case design
  6. Demonstrator evaluation plan
  7. Governance protocol
  8. Proof-of-concept implementation
  9. Final integrated package assembly
