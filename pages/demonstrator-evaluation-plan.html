Demonstrator evaluation plan

This page defines how the proof-of-concept demonstrator will be assessed. The goal is not to evaluate it as a production system, but to judge whether it successfully demonstrates the proposed method: bounded AI support for SMCE preparation work with traceability, human oversight, and visible uncertainty.

Evaluation purpose

The evaluation should answer three questions:

  1. Does the demonstrator show the method clearly and credibly?
  2. Does it produce usable artifacts with sufficient traceability and reviewability?
  3. Does it stay within the governance boundaries stated in the proposal?

Evaluation object

The evaluation covers:

  • the workflow shown in the demonstrator
  • the quality of generated artifacts
  • the clarity of human review and approval gates
  • the logging and traceability record
  • the handling of uncertainty, disagreement, and missing evidence
  • the usefulness of outputs for a policy-analysis context

It does not need to prove operational deployment readiness.

Core evaluation criteria

1. Traceability

Question: Can a reviewer reconstruct how an output was produced?

Look for:

  • clear links from evidence source to evidence sheet to rationale note to output
  • identifiable prompts, actions, revisions, and approvals
  • versioned artifacts
  • explicit references to affected criteria and options

Evidence to collect:

  • completed artifact set
  • logs or action trail
  • reviewer spot checks

Success standard:

  • a reviewer can trace a sample conclusion end to end without guesswork

2. Human oversight and approval

Question: Is human judgement visibly retained at the right points?

Look for:

  • named human roles
  • clear approval gates
  • draft status before approval
  • evidence of correction, rejection, or override where needed

Evidence to collect:

  • review log
  • approval notes
  • examples of accepted and corrected outputs

Success standard:

  • no important analytical output appears to have been accepted automatically

3. Output quality and usability

Question: Are the resulting artifacts clear and useful for officials?

Look for:

  • evidence sheets that are complete and readable
  • rationale notes that explain the move from evidence to judgement
  • uncertainty entries that support action
  • comparative summaries that are understandable and not misleading

Evidence to collect:

  • artifact review against templates
  • brief reviewer comments on usability

Success standard:

  • outputs are review-ready and require refinement, not reconstruction

4. Preservation of uncertainty and disagreement

Question: Does the method keep uncertainty visible instead of smoothing it away?

Look for:

  • explicit uncertainty entries
  • contested evidence marked as contested
  • provisional or caveated judgements where justified
  • evidence that disagreement is preserved in notes and outputs

Evidence to collect:

  • uncertainty register
  • rationale notes with caveats
  • examples of contradictory evidence handling

Success standard:

  • material uncertainty remains visible in the demonstrator outputs

5. Governance compliance with proposal boundaries

Question: Does the demonstrator do only what the proposal says AI may do?

Look for:

  • AI confined to support tasks such as structuring, drafting, gap flagging, and summary preparation
  • no evidence of AI making final policy judgement, final scoring authority, or hidden trade-off decisions
  • logs showing reviewable intervention points

Evidence to collect:

  • intervention-point map or equivalent workflow record
  • prompt and approval samples
  • evaluator review of demonstrator steps

Success standard:

  • demonstrator behaviour matches the governance-first, human-supervised design

6. Robustness under imperfect inputs

Question: Does the workflow still behave responsibly when evidence is incomplete, weak, or contradictory?

Look for:

  • visible downgrading of confidence
  • escalation or follow-up where evidence is insufficient
  • no unsupported certainty in outputs
  • appropriate use of uncertainty handling actions

Evidence to collect:

  • test cases using seeded evidence gaps and contradictions
  • reviewer notes on failure handling

Success standard:

  • weak inputs produce transparent caveats and review triggers, not false confidence

7. Value for the intended project deliverable

Question: Does the demonstrator generate outputs that can feed the integrated package?

Look for:

  • reusable artifact examples
  • clear methodological lessons
  • evidence that the demonstrator makes the framework tangible and testable

Evidence to collect:

  • mapping from demonstrator outputs to final deliverable sections
  • final evaluation memo

Success standard:

  • demonstrator outputs can be incorporated directly as examples, annexes, or tested procedures

Evaluation evidence set

At minimum, the evaluation should examine:

  • synthetic use case description
  • option and stakeholder definitions
  • sample completed evidence sheets
  • sample completed rationale notes
  • uncertainty register
  • intervention-point or action log
  • review-and-approval log
  • comparative summary output
  • evaluator comments and findings memo

Review approach

Use a structured review with three layers.

Layer 1. Artifact review

Check each artifact against its template and required fields.

Layer 2. Workflow review

Check whether steps occurred in the right order with visible review gates and logs.

Layer 3. Governance review

Check whether the demonstrator stayed within the allowed AI role and preserved human accountability.

Recommended evaluation method

  1. Select a sample path through the demonstrator from raw source to comparative output.
  2. Perform end-to-end traceability review.
  3. Check required fields and logs for each artifact type.
  4. Inspect at least one case of weak evidence and one case of conflicting evidence.
  5. Confirm where human review intervened.
  6. Record strengths, weaknesses, and required revisions.
  7. Decide whether the demonstrator is acceptable as a proof of method.

Suggested rating scale

Use a simple four-point scale for each criterion:

  • 3 = fully demonstrated
  • 2 = demonstrated with minor gaps
  • 1 = partly demonstrated, significant revision needed
  • 0 = not demonstrated

Optional summary categories:

  • Accept
  • Accept with revisions
  • Rework required

Evaluation record template

| Criterion | Rating (0-3) | Evidence reviewed | Main finding | Action needed | | --- | --- | --- | --- | --- | | Traceability | | | | | | Human oversight | | | | | | Output quality | | | | | | Uncertainty handling | | | | | | Governance compliance | | | | | | Robustness under imperfect inputs | | | | | | Deliverable usefulness | | | | |

Acceptance threshold

The demonstrator should normally be considered acceptable if:

  • no core criterion scores 0,
  • traceability, human oversight, and governance compliance each score at least 2,
  • any remaining gaps are documented and remediable without redesigning the whole method.

Typical review questions

  • Can we see where each important analytical statement came from?
  • Is there any point where AI output appears to have bypassed human validation?
  • Are weak evidence and uncertainty made visible?
  • Do the artifacts look like something officials could review efficiently?
  • Does the demonstrator help explain the method to JRC reviewers?
  • Are the limits of the demonstrator stated clearly?

Deliverables from the evaluation

The evaluation should produce:

  • completed evaluation table
  • short narrative findings memo
  • list of required revisions, if any
  • recommendation on inclusion in the integrated package
