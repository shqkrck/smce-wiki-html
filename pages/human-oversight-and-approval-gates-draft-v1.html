Human oversight and approval gates - draft v1

Document metadata

  • Document type: First-draft execution governance page
  • Status: Draft v1 populated from proposal commitments and working delivery assumptions
  • Scope: Human decision rights, review sequence, and approval controls for the AI-enabled SMCE methodology and demonstrator
  • Primary source set:

    • - AI-enabled SMCE technical proposal - Next steps after contract reread - Requirements traceability matrix - draft v1 - AI intervention-point register - draft v1 - Demonstrator evaluation plan

Why this page exists

The proposal repeatedly says that AI outputs must be human reviewed, but that is not yet enough to run a real project. This page turns that general commitment into:

  • named roles
  • decision rights
  • gate-by-gate approval criteria
  • rejection and rework rules
  • a proposed gate sequence for the actual package

Oversight baseline carried from the proposal

  • Every important AI output remains draft until human approval.
  • No analytical output can pass a gate without identifiable evidence support.
  • Uncertainty, disagreement, and missing evidence must remain visible at every gate.
  • Prompts/actions, revisions, and approvals must be logged.
  • Final analytical judgement, scoring, trade-offs, and policy decisions stay with humans.

Working role model for draft v1

These are function-based placeholders to be mapped to named individuals once the project team and JRC counterparts are confirmed.

Role R1. Analyst / preparer

  • Runs bounded AI-assisted drafting tasks.
  • Prepares source packs and artifact revisions.
  • Owns first-pass completeness.
  • Cannot self-approve substantive analytical artifacts for onward gate passage.

Role R2. Methodological reviewer

  • Checks traceability, methodological discipline, and boundary compliance.
  • Reviews evidence sheets, criteria structure, rationale notes, and uncertainty handling.
  • Can approve for next step, reject, or require rework.
  • Does not make final policy choice.

Role R3. Policy owner / case lead

  • Owns contextual fit of the synthetic case and the analytical narrative.
  • Decides whether unresolved issues are acceptable, deferred, or blocking.
  • Retains authority over substantive judgement, comparative framing, and any final scoring/trade-off steps outside AI scope.

Role R4. Validation sign-off role

  • Checks package completeness and governance readiness.
  • Confirms logs reconcile with artifacts.
  • Releases the integrated package to final acceptance review.

Role R5. JRC review counterpart

  • Participates at defined checkpoints.
  • Provides written comments and acceptance-oriented feedback.
  • Does not replace internal quality control; it sits on top of it.

Approval principles for this project

  • Preparation and approval should be separated for substantive outputs.
  • Approval is always version-specific.
  • Rejected versions remain visible in the record.
  • Conditional approval is allowed only for administrative cleanup, not substantive evidential gaps.
  • If evidence is missing or disagreement is material, the correct gate result is hold, rework, or explicit residual acceptance by R3, not silent passage.

Proposed gate sequence for the first real project package

  1. G-01 Scope and control baseline
  2. G-02 Synthetic case and source-pack readiness
  3. G-03 Evidence-sheet approval
  4. G-04 Criteria-structure approval
  5. G-05 Rationale-note approval
  6. G-06 Uncertainty and open-issues disposition
  7. G-07 Internal review-summary approval
  8. G-08 Demonstrator walkthrough and audit-pack approval
  9. G-09 Final integrated package release
  10. G-10 JRC comment incorporation and acceptance support

Gate outputs expected in the first package cycle

  • After G-01: stable control baseline pages
  • After G-02: frozen synthetic case v1 and approved source pack
  • After G-03: evidence sheets cleared for criteria and rationale drafting
  • After G-04: criteria baseline for demonstrator run
  • After G-05: rationale note set ready for summary drafting
  • After G-06: visible uncertainty posture and open-issues disposition
  • After G-07: checkpoint-ready internal summary
  • After G-08: acceptance-ready demonstrator trail
  • After G-09: submission-ready integrated package

Approval status model

  • Draft: created but not submitted for gate review
  • Under review: submitted to designated reviewer(s)
  • Rework required: failed current gate and must be revised
  • Approved for next step: may be used downstream
  • Accepted for package: locked for final integrated deliverable inclusion
  • Superseded: retained for traceability but no longer current
  • On hold: temporarily blocked by unresolved issue or missing evidence

Immediate rejection triggers

Reject at once if any of the following occurs:

  • evidence basis cannot be identified
  • AI output contains invented or unsupported content
  • a material caveat or disagreement has been smoothed away
  • logging is missing for an artifact being presented as gate-ready
  • AI has crossed into final scoring, weighting, ranking, or policy recommendation
  • artifact version used downstream is not the approved version

Standard rework flow

  1. Reviewer records reject or rework with explicit reasons.
  2. Artifact owner revises the content and, if needed, the associated logs.
  3. New version ID is assigned.
  4. Reviewer checks only the newest version for decision, while keeping earlier versions visible.
  5. If the same failure repeats, R3 decides whether this is a narrow artifact problem or a method-control problem requiring register updates.

Conditional approval rule

Conditional approval is allowed only when:

  • the missing item is administrative or formatting-related
  • the evidential basis is already adequate
  • owner and due date are logged
  • the condition is closed before the artifact becomes Accepted for package

Logging fields required at every gate

  • Gate ID
  • Artifact ID and version
  • Decision date/time
  • Reviewer/approver role and name if available
  • Decision outcome
  • Reason for reject/rework/hold if applicable
  • Linked issue or uncertainty IDs
  • Next permitted use of the artifact
  • Link to AI prompt/action and revision logs where relevant

Known unresolved items

  • Named individuals are not yet assigned to R1-R5.
  • It is not yet fixed which gates require live JRC participation and which are internal only.
  • Approval tooling is not yet selected; current assumption is file-based logs plus versioned artifacts.
  • Whether JRC will want a formal sign-off memo, tracked email confirmation, or both is still unknown.
  • If the synthetic case changes significantly, G-02 and possibly G-04 criteria approval will need to be rerun.
