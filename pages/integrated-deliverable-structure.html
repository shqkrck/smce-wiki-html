Integrated deliverable structure

This page defines the shape of the final integrated package to be delivered under the AI-enabled SMCE contract. It translates the proposal requirement for one integrated deliverable into an operational package structure that is easy to assemble, review, and submit.

Purpose

The final delivery must work as one coherent package that:

  • explains the method
  • shows the governance rules
  • demonstrates the method on a synthetic case
  • records how outputs were reviewed
  • provides enough annexed evidence for auditability

The package should read as a controlled methodological deliverable, not as a loose set of disconnected notes.

Packaging principles

The integrated deliverable should be:

  • method-first rather than technology-first
  • explicit about allowed and prohibited AI roles
  • traceable from requirement to artifact
  • readable by non-technical reviewers
  • suitable for electronic submission in English
  • structured so that the demonstrator supports, rather than distracts from, the methodology

Recommended top-level structure

Cover and front matter

Include:

  • title page
  • version and date
  • authorship or contractor identification as appropriate
  • document control table
  • table of contents
  • glossary or abbreviation list if needed
  • short executive summary

Part A. Methodological framework

Purpose: explain the overall approach.

Recommended contents:

  1. purpose and scope
  2. project context and intended use
  3. relationship to SMCE and EC impact assessment workflow
  4. core workflow stages
  5. AI intervention points
  6. human roles and approval gates
  7. artifact set and how artifacts interact
  8. traceability and logging approach

Main supporting pages or artifacts:

  • AI-enabled-SMCE-technical-proposal
  • requirements traceability matrix
  • intervention-point register
  • human oversight and approval gates

Part B. Governance and control protocol

Purpose: make the governance claims testable and auditable.

Recommended contents:

  1. what AI may do
  2. what AI may not do
  3. draft status rule for AI outputs
  4. evidence traceability rule
  5. uncertainty visibility rule
  6. review and approval requirements
  7. logging and versioning requirements
  8. escalation and non-compliance handling

Main supporting artifacts:

  • governance rules table
  • prompt/action logging approach
  • review-and-approval log

Part C. Core templates and working artifacts

Purpose: provide the reusable operational backbone.

Recommended contents:

  1. artifact set overview
  2. instructions for use
  3. completed or sample templates

Minimum template set:

  • Evidence sheet template
  • criteria structure template
  • Rationale note template
  • Uncertainty register template
  • intervention-point log
  • review-and-approval log
  • final internal summary template

Part D. Synthetic use case

Purpose: define the demonstrator case and why it is suitable.

Recommended contents:

  1. use case overview
  2. policy problem and baseline
  3. option set
  4. stakeholder set
  5. evidence corpus description
  6. designed uncertainties and conflict points
  7. review checkpoints
  8. reasons this case is a good methodological test

Main supporting page:

  • Synthetic use case design

Part E. Demonstrator walkthrough

Purpose: show the method in action.

Recommended contents:

  1. walkthrough scope and limitations
  2. step-by-step demonstrator sequence
  3. example intervention points
  4. example evidence-sheet creation
  5. example rationale-note creation
  6. example uncertainty handling
  7. example review gate and approval
  8. resulting comparative summary

Include representative screenshots, logs, or output extracts if appropriate.

Part F. Demonstrator evaluation and lessons

Purpose: show how the demonstrator was assessed and what was learned.

Recommended contents:

  1. evaluation criteria
  2. evidence reviewed
  3. findings by criterion
  4. strengths
  5. weaknesses and limitations
  6. recommended refinements
  7. implications for any future operational use

Main supporting page:

  • Demonstrator evaluation plan

Annexes

Purpose: preserve auditability without overloading the main narrative.

Recommended annexes:

  • requirements traceability matrix
  • intervention-point map or register
  • completed sample artifacts
  • uncertainty register extract
  • review-and-approval log
  • action or prompt log extract
  • bibliography and source list
  • terminology note if needed

Required cross-links inside the package

The integrated deliverable should make these chains easy to follow:

  • requirement -> design response -> artifact -> evidence of compliance
  • source -> evidence sheet -> rationale note -> matrix or summary output
  • uncertainty -> handling action -> reporting statement
  • AI action -> human review -> approval or correction

Minimum quality checks before package finalisation

Before final assembly, confirm that:

  • every major claim in the methodology is backed by an artifact, rule, or demonstrator example
  • governance rules are stated in operational language
  • synthetic use case materials are complete enough to understand the walkthrough
  • demonstrator evaluation findings are included
  • annexes contain enough sample evidence for reviewer spot checks
  • document structure is coherent and not repetitive

Suggested assembly order

Build the final package in this order:

  1. lock structure and headings
  2. finalise methodological framework text
  3. finalise governance protocol
  4. insert core templates and usage notes
  5. insert synthetic use case description
  6. insert demonstrator walkthrough
  7. insert evaluation findings
  8. attach annexes and traceability material
  9. run final consistency and cross-reference check

What should remain out of scope

Unless separately requested, the integrated deliverable should not imply:

  • production deployment readiness
  • full system integration with EC infrastructure
  • autonomous AI decision-making
  • replacement of official judgement or formal policy choice

Practical takeaway

The integrated deliverable should let a reviewer understand, in one package, both the method and the proof that the method was exercised in a controlled, reviewable, governance-compliant way.
