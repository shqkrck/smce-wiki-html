V2 refinement plan

Document metadata - Document type: Execution control plan - Status: Working plan for transition from draft v1 to client-reviewed v2 - Purpose: Define the concrete refinements, confirmations, decisions, and drafting actions required before the package is suitable for JRC-facing review - Primary source set: - Requirements traceability matrix - draft v1 - AI intervention-point register - draft v1 - Human oversight and approval gates - draft v1 - Synthetic use case proposal - draft v1 - Demonstrator evaluation plan - draft v1 - Integrated deliverable outline - draft v1 - JRC decision log - Reviewer questions list

Objective of this plan

This plan converts the populated draft set into a controlled route to v2. It is not a generic editing checklist. It is the working control document for: - resolving contradictions and open assumptions in v1; - deciding what must be confirmed with JRC versus what can be refined internally; - sequencing the minimum work needed to produce a coherent v2 review pack; - preventing hidden ambiguity from leaking into the client review cycle.

V2 success condition

The package is ready for client-reviewed v2 when all of the following are true: - the synthetic use case is confirmed and used consistently across all pages; - the intervention points, approval gates, and deliverable outline use aligned terminology and artifact names; - major open decisions are either resolved or clearly flagged as JRC decision requests; - the package can be reviewed as one coherent methodology-plus-demonstrator proposal rather than disconnected control pages; - reviewer questions are consolidated into a short, answerable set tied to concrete consequences; - a practical v2 review pack can be circulated without needing the reader to infer missing control logic.

What v1 already establishes

The draft set already provides a strong execution baseline: - a populated requirements-to-artifacts control spine; - a bounded AI intervention map with allowed and prohibited tasks; - a gate model with human decision rights and rejection triggers; - a synthetic use case with options, criteria, corpus design, and seeded uncertainties; - a demonstrator evaluation logic with criteria and acceptance thresholds; - an integrated submission outline with parts, annexes, and assembly sequence.

Critical findings from the v1 review

1. Synthetic use case alignment has been achieved provisionally, but JRC confirmation is still required

The draft package is now internally aligned on the household energy-support access case as the working planning anchor.

This remains the single most important item requiring JRC confirmation because it affects: - the demonstrator narrative; - the seeded evidence corpus; - criteria wording; - uncertainty examples; - the integrated package Part D description; - the reviewer question set.

2. Role placeholders are workable but not yet review-ready

The role model R1-R5 is clear enough for internal use, but v2 should decide whether JRC should see: - named individuals; - named functions only; or - functions in the body with named owners in the review pack cover note.

3. Several acceptance-critical controls are still expressed as assumptions

The following items remain unresolved and should not remain implicit in v2: - final delivery packaging format; - log storage and retention posture; - citation granularity for traceability; - expected JRC review cadence and which gates involve JRC directly; - demonstrator environment and tool/model posture.

4. The draft set is strong on governance but needs a sharper client-facing control layer

v1 is content-rich, but v2 should make it easier for JRC to answer key questions quickly. That means: - fewer buried assumptions; - tighter cross-references; - clearer decision requests; - a visible distinction between settled design and requested confirmation.

V2 workstreams

WS-1. Baseline alignment and contradiction removal

Goal: - eliminate internal inconsistency before external review.

Required actions: 1. Confirm the synthetic use case to carry into v2. 2. Update all cross-references, examples, and seeded uncertainties so they point to the same case. 3. Check artifact names and links across all v1 pages for consistent page titles. 4. Harmonise wording for review roles, gates, logs, and artifact IDs.

Primary dependencies: - JRC decision on use case, or internal provisional choice pending JRC review.

Outputs for v2: - internally consistent package narrative; - explicit note where a remaining assumption is still pending confirmation.

WS-2. JRC decision capture and escalation control

Goal: - turn open ambiguity into visible decision requests.

Required actions: 1. Move all material unresolved items from the v1 pages into JRC decision log. 2. For each decision, record options, recommendation, owner, impact, and target timing. 3. Separate true JRC decisions from internal drafting choices. 4. Mark which decisions block v2 circulation and which can remain open during review.

Outputs for v2: - one decision log that can be reviewed before or during the first v2 checkpoint.

WS-3. Reviewer question pack preparation

Goal: - give JRC a short, structured set of questions that directly unblock progression.

Required actions: 1. Consolidate all confirmation requests from v1 into grouped questions. 2. Phrase questions so they are answerable in a meeting or short written review. 3. Tie each question to the impacted artifact, gate, or delivery consequence. 4. Distinguish questions for confirmation, preference, and decision.

Outputs for v2: - Reviewer questions list ready for circulation with the v2 pack.

WS-4. Method-control hardening

Goal: - make the governance model easier to review and harder to misread.

Required actions: 1. Confirm how version naming will work in practice. 2. Confirm minimum citation granularity and traceability standard. 3. Confirm whether stakeholder-input coding becomes a separate intervention point. 4. Confirm whether uncertainty drafting remains separate from gap flagging. 5. Add standard disclaimer language for draft AI-assisted outputs.

Outputs for v2: - clearer operational rules in the control pages; - reduced risk of reviewer confusion over AI boundaries.

WS-5. Deliverable assembly readiness

Goal: - ensure the v2 pages can feed the integrated package cleanly.

Required actions: 1. Align the selected use case and artifact set with Part D and Part E of Integrated deliverable outline - draft v1. 2. Confirm whether final delivery is one compiled document plus annexes or another bundle structure. 3. Clarify which sample artifacts are expected inline versus in annexes. 4. Confirm the minimum review pack composition for the first client-facing cycle.

Outputs for v2: - cleaner path from v2 review pages into the final integrated package.

Required confirmations before v2 circulation

These are the minimum confirmations that should be sought before the first JRC-facing review pack is treated as stable.

  1. Which synthetic use case does JRC want reviewed?
  2. Does JRC want to review at the scope baseline, use-case baseline, or only once the first integrated v2 pack exists?
  3. What final delivery packaging does JRC prefer?
  4. What level of tool/model specificity is expected in the demonstrator description?
  5. Are log extracts sufficient for review, or does JRC expect the full underlying trail annexed?

Proposed v2 review pack contents

The first client-reviewed v2 pack should contain at minimum: - refined control pages for requirements, intervention points, and approval gates; - one confirmed synthetic use case page; - one decision log; - one reviewer questions list; - one short cover note stating what is settled, what needs confirmation, and what will change after review.

Recommended execution sequence

  1. Resolve the use-case inconsistency.
  2. Freeze the decision log and reviewer question list.
  3. Apply terminology harmonisation across the control pages.
  4. Confirm packaging, log, and review-cadence assumptions.
  5. Prepare the v2 review pack and circulate it for JRC comment.
  6. After review, update the integrated deliverable outline and downstream demonstrator materials accordingly.

Entry / exit criteria

Entry to v2 refinement

  • v1 draft pages exist and are materially populated.
  • unresolved issues are visible enough to convert into decisions and questions.

Exit from v2 refinement

  • one consistent synthetic case is used throughout;
  • all major open items are either resolved or explicitly parked in the decision log;
  • reviewer questions are consolidated and prioritized;
  • no critical contradiction remains across the core control pages;
  • the package is suitable for JRC review without hidden internal assumptions.

Risks to manage during refinement

  • keeping both synthetic cases alive too long will create compounding inconsistency;
  • overloading JRC with low-value questions may slow the first review loop;
  • deferring packaging and log decisions too long may create late-stage assembly rework;
  • adding too much detail to intervention points may make the demonstrator look more complex than intended;
  • leaving role naming ambiguous may weaken confidence in oversight design.

Immediate next actions

  • Use JRC decision log as the single control point for unresolved external decisions.
  • Use Reviewer questions list as the client-facing prompt sheet for the first review loop.
  • Update the affected v1 pages only after the use-case and review-cadence decisions are clarified.
