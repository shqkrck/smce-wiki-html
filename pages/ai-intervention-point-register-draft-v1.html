AI intervention-point register - draft v1

Document metadata

  • Document type: First-draft execution control page
  • Status: Draft v1 populated from proposal commitments
  • Scope: Bounded AI support within the SMCE preparation workflow and synthetic demonstrator
  • Primary source set:

    • - AI-enabled SMCE technical proposal - Next steps after contract reread - Requirements traceability matrix - draft v1 - Human oversight and approval gates - draft v1 - Evidence sheet template - Rationale note template - Uncertainty register template

Why this draft exists

The proposal commits to explicit AI intervention points. This page turns that commitment into an operational register that can actually run the demonstrator. It names:

  • where AI enters the workflow
  • what the AI is allowed to do
  • what it is not allowed to do
  • who reviews the output
  • what artifact is created
  • what must be logged
  • what triggers escalation

Working operational assumptions behind this register

  1. The demonstrator uses one synthetic Better Regulation-style case, currently assumed to be the household energy-support access case described in Requirements traceability matrix - draft v1 and Synthetic use case proposal - draft v1.
  2. All AI outputs are draft-only until approved.
  3. The demonstrator is file-based. Every intervention should leave a reviewable artifact and log entry.
  4. Human scoring, weighting, ranking, trade-off resolution, and final policy framing are out of AI scope.
  5. The intervention map is intentionally small because the proposal promised a small number of explicit, auditable intervention points.

Core roles used in this register

  • Analyst / preparer: runs the step, checks immediate fidelity, and assembles artifacts.
  • Methodological reviewer: checks traceability, methodological fit, and compliance with AI boundaries.
  • Policy owner / case lead: decides whether the artifact is fit for use in the synthetic case and whether unresolved issues are tolerable.
  • Validation sign-off role: checks that only approved versions and complete logs enter the final package.

Named artifacts referenced in this register

  • SRC-REG: Source register
  • EV-SET: Evidence sheet set
  • CRIT-01: Criteria structure note
  • RAT-SET: Rationale note set
  • UNC-REG: Uncertainty register
  • SUM-01: Internal review summary
  • BRIEF-01: Checkpoint briefing note
  • LOG-AI-01: Prompt and action log
  • LOG-REV-01: Review and approval log
  • DEMO-01: Demonstrator walkthrough script
  • AUDIT-01: Audit pack

Global control rules

  • All AI outputs must be labelled draft at creation.
  • No substantive statement may survive review without source traceability.
  • Weak, contradictory, or missing evidence must create visible caveats or uncertainty entries.
  • If an intervention crosses into prohibited judgment territory, the output is rejected rather than corrected silently.
  • Every intervention must be replayable from artifact ID + log ID + reviewer decision.

Minimum log fields for every intervention

  • Intervention ID
  • Run date/time
  • Operator
  • Input artifact version(s)
  • Tool/model used
  • Prompt or action summary
  • Output artifact version
  • Reviewer role/name
  • Review outcome
  • Issues found
  • Escalation trigger Y/N
  • Linked evidence IDs
  • Linked uncertainty IDs if any

Planned intervention sequence for the demonstrator

  1. IP-01 source ingestion and triage
  2. IP-02 evidence-sheet drafting
  3. IP-03 criteria-structure drafting
  4. IP-04 rationale-note drafting
  5. IP-05 gap and inconsistency flagging
  6. IP-06 uncertainty-entry drafting support
  7. IP-07 internal review-summary drafting
  8. IP-08 checkpoint brief preparation
  9. IP-09 audit-pack consolidation

Provisional seeded uncertainties the register is expected to handle

  • U-01: Take-up gains from simplified or proactive enrolment may be lower than expected once real verification frictions are considered.
  • U-02: Automatic or data-assisted eligibility checks may improve coverage but create legal, privacy, or due-process concerns for some groups.
  • U-03: Administrative savings for public authorities may be offset by data-matching complexity, appeals handling, or local implementation burdens.
  • U-04: Stakeholder submissions may mix affordability, fairness, privacy, administrative burden, and implementation feasibility arguments without separating criteria cleanly.
  • U-05: Legal feasibility of proactive or semi-automatic enrolment may require narrower data-sharing or review safeguards than the synthetic option text assumes.

Prohibited tasks across the whole intervention map

  • Final scoring or weighting by AI
  • Final ranking of options by AI
  • Autonomous trade-off resolution
  • Replacement of stakeholder consultation with model-generated stakeholder views
  • Use of live EC operational files in the demonstrator
  • Inclusion of unlogged AI-generated substantive content in the final package

Planned sample artifact flow

Synthetic source -> SRC-REG -> EV-SET -> CRIT-01 -> RAT-SET -> UNC-REG -> SUM-01 -> BRIEF-01 -> DEMO-01/AUDIT-01

Exit conditions for each intervention

An intervention is complete only when:

  • the output artifact has a version ID
  • the log record is complete
  • the designated reviewer has recorded approve, reject, or rework
  • any material uncertainty or issue is linked forward
  • the next intervention is told exactly which artifact version it may use

Main unresolved items

  • Whether stakeholder-input coding should be a separate intervention point or embedded in IP-02.
  • Whether uncertainty drafting should remain a distinct IP-06 or be absorbed into the gap-flagging stage.
  • Which model/tool will be used for the demonstrator and whether it supports exportable run logs automatically.
  • Whether JRC wants one checkpoint brief format or separate technical and managerial briefing variants.
